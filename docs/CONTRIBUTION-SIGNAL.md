# kg-hub 胶囊「贡献度」自动测量 — 专项设计

> **Status**: 📐 设计中（2026-06-22 提出）。未实现。
> **关联**: DESIGN.md 已知局限 L1；排序修复 capsule-usage-audit-2026-06-18。

## 1. 问题陈述

`usage_count` 测的是**曝光**（胶囊被 PUSH hook 注入了多少次），不是**贡献**（它是否真的影响了 agent 的输出 / 帮用户解决了问题）。

期望：**自动**测贡献度，不依赖 human 判断。因为 human 并不知道——
- 某次会话实际"引用"了哪篇胶囊；
- 那篇胶囊的内容是什么；
- 它对最终输出是否有真实贡献。

## 2. 为什么这很难

- **闭环缺失**：`注入 → agent 是否读了 → 是否改变输出 → 是否有益`，目前一个反馈都没采。
- **LLM 归因本质糊**：上下文里某段是否"导致"了某个输出，难以干净地证明因果。
- **真贡献 = 反事实**：严格定义是"不注入它，输出会不会不同"。在线拿不到（不能每次都跑两遍）。

**现实目标**：不是求"因果真值"，而是造一个**远好于曝光的自动代理**，并始终明确它仍是代理。

## 3. 分层方案（按信号强度 / 成本，建议自底向上落地）

### Tier 1 — 参与度检测（便宜、确定性、先上）

会话结束后（复用现有 15-min ingester），把**本会话注入过的胶囊**与**本会话产生的 observations / diff / 工具调用**做重叠检测：
- 实体 / 专有名词 / 独特短语重合；
- 会话内是否显式 `kg_search` / 打开了该胶囊；
- 是否复用了该胶囊独有的命名（函数名、配置键、路径）。

→ 产出 `engagement_count`（与 `usage_count` 并列）。**立刻能看到"曝光 vs 参与"的裂口**（例：DESIGN 注入 68 次、参与 3 次 = 多数注入是噪音）。

- **优点**：纯确定性，一个脚本，零 LLM 成本。
- **局限**：相关非因果（可能"刚好聊到"而非"因为注入才聊到"）。

### Tier 2 — LLM 裁判贡献分（自动、中等成本）← 「human 出局」那一层

ingester 里对每个（注入胶囊 × 会话）加一次 cheap（Haiku 级）调用：

> 输入：[注入的胶囊正文] + [会话实际做了什么：observations / 结果 / diff]
> 任务：打分 0-2（0=忽略 / 1=上下文相关 / 2=直接使用），**必须指出具体重合证据**，无据则默认 0。

→ 产出 `contribution_score`。这是最直接的自动贡献代理。

- **优点**：自动、无 human、信号比 Tier 1 强。
- **局限**：裁判有偏 / 谄媚，需用 Tier 3/4 校准；默认判 0、强制要证据可压低假阳。

### Tier 3 — 反事实 / 消融（最强、最贵 → 离线抽样）

金标准：同任务"注入 vs 不注入"各跑一遍，diff 输出差异。在线不可行。
离线：每周抽 N 个历史会话，消融该胶囊后重放，测输出发散度。给**真因果信号**，用来**校准**便宜的 Tier 1/2（确认它们没系统性跑偏）。

### Tier 4 — 把「探索槽」当自然实验（自动、统计）

**关键洞察：探索槽已经在随机地「注入 / 不注入」长尾胶囊——这本身就是一个免费的 A/B！**

只需定义一个**自动的会话产出指标**，对比"拿到该胶囊 vs 没拿到"的会话指标差，估计该胶囊的平均 lift。随机化是免费的（探索已上线），唯一缺的是产出指标。

## 4. 共同关键：自动产出指标

所有自动法都卡在「每会话的质量 / 成功信号」。全自动候选：
- **参与**：agent 是否引用了胶囊内容（Tier 1 本身）；
- **成功代理**：无错误回环、用户未纠正 / 未重述同一请求、任务闭合；
- **隐式用户信号**：输出被接受、未被改写；
- **下游复发**：同类问题是否再次发生——尤其对 INCIDENT-RETRO 这类，若它**防止了复发**，就是真贡献。

## 5. 建议落地顺序

1. **Tier 1 `engagement_count`**（确定性脚本，复用 ingester + claude-mem observations）→ 立刻暴露曝光/参与裂口。
2. **Tier 2 `contribution_score`**（ingester 加一次 Haiku 裁判）→ 真正的自动贡献代理。
3. **排序信号从 `usage_count` 切到 `contribution_score`**；`usage_count` 降级为"曝光记账"。
4. **Tier 4** 用探索随机化做总体校准（可选，验证 Tier 1/2 不跑偏）。
5. 文档明确：这仍是**代理**，不是因果真值；Tier 3 消融才是真值，按需抽样。

## 6. 现成可复用的件（为什么这不是从零造）

- **claude-mem 每会话 observations**（已采）→ Tier 1 重叠检测的输入。
- **15-min ingester cron + graphiti 的 Anthropic LLM client**（已有）→ Tier 2 裁判挂这。
- **探索槽随机化**（2026-06-18 已上线）→ Tier 4 的免费 A/B 基础设施。
- **`usage_count` / `last_used_at` 字段 + bump 路径**（已有）→ 加 `engagement_count` / `contribution_score` 是同构扩展。

## 7. 开放问题

- 重叠检测的阈值与去噪（停用词、通用术语会制造假阳）。
- Tier 2 裁判的成本上限（每会话注入 3 篇 → 每会话 3 次 Haiku；按量评估）。
- "成功代理"指标本身也是代理，可能引入新偏差——需 Tier 3 兜底校准。
- 胶囊正文常超 2000 字，裁判 / 重叠用全文还是摘要，影响成本与准确度。

## 8. 实测发现（2026-06-22，Tier-1 + Tier-2 首跑）

工具：`tools/engagement_audit.py`（Tier-1）、`tools/contribution_judge.py`（Tier-2）。

- **Tier-1（术语重合）**：OBSERVATION-PHASE 参与率 73%、ONBOARDING 42%，而 **DESIGN 曝光 74 但参与率 0%** —— 首次把「曝光≠贡献」量化出来。局限：只认逐字术语，对语义复述失明（DESIGN 的设计词汇很少在编码 observations 里逐字重现）。
- **Tier-2（LLM 裁判）暴露两个真问题**：
  1. **prompt 敏感 / 话题混淆**：初版 prompt 给 DESIGN 均值 1.33（证据竟是「这是最常被注入的文档」——拿曝光当贡献的假阳）；收紧 prompt（禁止「同话题=有贡献」、要求胶囊具体点↔会话动作一一对应、拒绝元描述）后，**同批数据全部归零**。结论：**Tier-2 单独不可作真值，对框架极敏感**。
  2. **样本偏置**：首跑的 25 对几乎全是「分析排序系统本身」的元工作会话，本就没在「使用胶囊内容做别的任务」，故 0 合理——但**不能据此判定胶囊无用**（有偏小样本 + 易摆动裁判 → 过度外推）。
- **Tier-1 与 Tier-2 在 DESIGN/OBSERVATION-PHASE 上结论几乎相反**：不是 bug，是两者各抓一种「使用」（逐字复用 vs 语义相关）。**分歧点正是 Tier-3 消融该裁决的地方**。

**修正后的落地判断**：Tier-1 可直接长期跑（确定性、便宜、暴露曝光裂口）；Tier-2 用收紧 prompt 作辅助交叉验证，但**在拿到代表性样本 + Tier-3 校准前，两者都不应直接驱动排序**。当前排序继续用「相关性 + 探索」。
